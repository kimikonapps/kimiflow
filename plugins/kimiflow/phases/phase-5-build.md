## 🟠 Phase 5 — Implement / fix

**Default: 1 `balanced` implementation subagent, sequential.** Full tools, fresh context, inputs `PLAN.md` + `ACCEPTANCE.md` + `INTENT.md`/`PROBLEM.md` (+ `DIAGNOSIS.md`). Promote the seat to `top` for risky, tightly coupled, architecture-shaping, or repeatedly failing work. `cheap` is never the normal implementer; it may only perform a deterministic, bounded support task with an explicit oracle.

- **TDD where sensible:** failing test first (Red) → commit tests before the implementation → green → refactor. In schema 4+ the Red test commit is an authorized early atomic checkpoint: test files only, explicitly named paths, announced in one line — production code never rides along. Before that commit, inspect the staged test-only diff and run `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/test-weakening-scan.sh` plus the bundled secret/path hygiene; append and resolve every `FLAG` in `ADVISORIES.md`. Inventory any foreign staged paths, leave them untouched, and use the path-limited commit procedure from reference.md. Schema 3 treats it as the legacy commit-gate exception (→ reference.md "Commit hygiene"). In fix mode the reproduction is the Red test and `BUG-REPRO.md` records Red command/status before code changes, then Green command/status plus regression evidence after the fix. Address the cause, not the symptom.
- **Clean-tree verification checkpoint (schema 4+, conditional):** when the decisive local build/test demonstrably refuses a dirty worktree, do not ask for another confirmation. Preserve the active session's `started_head` as the later review base; stage only named run-owned paths, inventory but never alter foreign staged paths, and inspect status/staged diff. Before committing, run `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/test-weakening-scan.sh` plus the bundled secret/path hygiene over that staged diff, append/triage every `FLAG` in `ADVISORIES.md`, and run every verification that can operate dirty. Then create one announced path-limited local `verify:` checkpoint under the original build authority and run the clean-tree-only check immediately. Failure returns to Phase 5; fix and create the next named local checkpoint rather than stopping or amending foreign history. This exception exists only to unlock the decisive verifier; normal paths remain uncommitted until Phase 7. Push/publication remain separately authorized. → reference.md "Commit hygiene".
- **Surgical:** every changed line traces to plan/intent/diagnosis. Leave foreign code alone, clean your own orphans. Every deletion carries a caller-grep proving zero callers (→ reference.md "Code mandate"); no proof → don't delete.
- **Escalation keyed on the FIRST failure's signal** (failure-stop): a clear test/stack failure → one targeted execution-feedback fix; an unclear / unexpected-API / likely-guess failure → escalate to research immediately (`WebSearch`/context7) — don't burn a blind second attempt. After two failed fix attempts, hand the failure evidence (failing command+output, diff, `DIAGNOSIS.md` path) to a **cross-family diagnosis call** when available (bounded, → reference.md "Model routing (per-role)"); its hypothesis is candidate-only — verify before applying. After repeated technical failure, change the hypothesis/approach/architecture and continue through the active-session Stop gate. Never ask merely to run another technical iteration; only a schema-4+ material decision kind may pause.
- **Adaptive execution (Contract 1 only):** follow `status.transition.execution`: the quality profile stays `compact|standard|critical`, strategy is independently `normal|recovery`, and `hard` budget pressure means prune optional breadth—not required gates, tests, evidence, or correctness. The owner Stop hook counts the turn automatically. Record a new decisive run artifact only once with `active-run.sh observe --event build|verification|review --outcome progress|passed|failed --evidence .kimiflow/<slug>/<artifact> --write`; never record prompts, raw source churn, comments, whitespace, or repeated reads as progress. A `no_progress` action changes the current phase's strategy and continues autonomously.
- **Convergence slices (Contract 1 only):** implement the PLAN's `S1..SN` in dependency order and run each exact `Check Sn` as soon as that slice is complete. A failed slice check is repaired before widening the diff. If the evidence reproduces a critical `Failure class Fn`, apply its declared semantic `Reset Fn` and rerun the smallest affected slice/falsifier instead of spending another broad review round. Do not create commits per slice unless the existing Red or clean-tree checkpoint rule requires one.
