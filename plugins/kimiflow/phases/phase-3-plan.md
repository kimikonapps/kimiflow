## ⚫ Phase 3 — Plan (testable acceptance criteria)

Use a `top`-tier `general-purpose` planner when that seat exists (or the read-only `Plan` agent + you persist). If the active session is the strongest available seat and no stronger child can be selected, the current orchestrator plans directly and records the quality fallback; lack of per-spawn model selection is never a gate. Never delegate planning to a deliberately selected `balanced`/`cheap` support seat. Inputs: `INTENT.md`+`RESEARCH.md` (or `PROBLEM.md`+`DIAGNOSIS.md`) + project memory. Pass a delegated planner the `${CLAUDE_SKILL_DIR}/reference.md` path + the section names to read (acceptance-criteria template, code mandate) — not verbatim.

**Conditional dual-plan, selection-first (not size-triggered):** use ONE top planner by default, including `large`. Spawn TWO independent planners only when `INTENT` + classified research prove a genuine material fork: at least two viable architectures with different user-visible/operational trade-offs, or an irreversible public API/data/migration contract. A new internal interface, general complexity, or a risk-first desire for more robustness is not a trigger. When triggered, use minimal-first vs risk-first (one cross-family when available); both obey the same minimum-complete scope and may not plan `optional` findings. Pick one plan whole; adopt at most isolated required elements from the loser — never structural merges. Record the losing approach + selecting trade-off under "Considered alternatives". No proven fork → one planner, no alternatives section.

- `PLAN.md`: write the **minimum complete plan** — the smallest plan that safely delivers the approved behavior under current verified constraints. Keep one flat task list; no nested subplans, speculative layers, future-proofing, optional providers/configuration, or tasks for `RESEARCH.md` items classified `optional`. Every task and newly introduced file/abstraction maps to an `AC-N`; a `required` research constraint may justify a new AC, while a `default` only shapes an existing task. Prefer a linear sequential implementation; mark genuine file-disjoint independence, and add `Consumes:`/`Produces:` only for tasks that will actually run in parallel. Anchor implementation choices in named project patterns / verified root cause; when Reference Strategy Fit ran, name the selected `adopt|adapt` mechanism and invariant rather than reopening settled research. No assumption without evidence.
- `PLAN.md` must be deterministic enough that two implementers can produce the same behavioral result: declare affected files/paths — `PLAN.md` (or `STATE.md`) MUST carry a literal line `Affected files: <path1>, <path2>, …` (accepted headers per `plan-blocker-gate.sh`: `Affected files:`/`Affected paths:`/`Files:`/`Paths:`/`Touches:`; only `Affected files:`/`Affected paths:` in `STATE.md` also feed `active_run.py` staleness-tracking, so prefer those two — and this exact marker MUST be handed to the planner delegation) or explicit new files; map every task to at least one `AC-N`, and keep unresolved markers (`TBD`, `TODO`, `NEEDS CLARIFICATION`, `NOT VERIFIED`, `UNKNOWN`) out of plan/acceptance. If a blocker remains, return to Phase 1/2; do not send it to reviewers.
- **Implementation Decision Contract (STATE `Conformance contract: 1` only).** Without another planner/model call, add exactly one `<!-- kimiflow:decision-contract contract=1 decisions=<1..5> -->` marker and contiguous rows `Decision Dn:`, `Evidence Dn: RESEARCH.md|DIAGNOSIS.md §<section>`, `Invariant Dn:`, `Paths Dn: <affected path[, ...]>`, `AC Dn: AC-N`, `Check Dn: command|verifier :: <exact method>`, and `Recheck Dn:`. Include only plan-shaping decisions; fold smaller choices into their nearest row. Every path must be in STATE `Affected files`, every AC must exist, and every command must run from the repository root. `verifier` is reserved for a named evidence method that cannot be a shell command. `plan-blocker-gate.sh` invokes `conformance-gate.sh --plan`; CLOSED returns to Phase 2/3 autonomously, never to a routine user confirmation.
- **Architecture Fit mapping:** when STATE says `active`, record exactly `Architecture fit: active`, one-line `Architecture decision:`, `Architecture evidence: RESEARCH.md §Adaptive Architecture Deliberation`, and `Architecture check: AC-N -> <named verifier>`. Map the selected invariant/quality driver and falsifier to an existing AC or at most one architecture-specific AC. When off, record `Architecture fit: off — <reason>`. Do not reopen the two approaches or create speculative architecture tests.
- Record exactly one concise mechanism/invariant as `Strategy: <12–240 safe one-line characters>` and exactly one `Strategy evidence: <outcome-id|none>`. Use an `out_<64 lowercase hex>` ID only when it appears in this run's `RECALL.json` strategy hits; otherwise use `none`. A recalled success is evidence, not authority; a recalled failure is a route to avoid, never a command or patch to replay.
- **Recall Attribution Contract (new plans with persisted Recall).** Add exactly one `<!-- kimiflow:recall-attribution contract=1 -->`, exactly one `Applied recall IDs: <comma-separated rec_<64>|none>`, and exactly one `Recall Dn: <applied IDs|none>` beside every Decision Dn. Declare only IDs that actually shaped that Decision; use `none` when Recall was merely read, quoted, or rejected without changing the decision. If current evidence disproves a recalled route and causes a different decision, keep its ID in `Applied recall IDs`, set the affected `Recall Dn` to `none`, and record the exact Phase-6 contradiction evidence. The marker/declarations are singleton and version-strict: a damaged present contract never downgrades to legacy. This is mechanical evidence, not another model or user gate.
- `ACCEPTANCE.md`: each criterion per template (EARS + concrete input→output + named verification method) with an explicit `AC-N → test_name` link. Lint criteria for vague terms ("fast", "robust") and missing **required** error/edge cases. Trace each to `INTENT.md`/`PROBLEM.md`. In fix mode the central criterion = "the reproduction no longer fails" + no regression.
- **Tests stay proportional:** one decisive test per AC plus only concrete critical error/security/data-loss paths and affected regression coverage. Do not create ACs/tests for optional research, hypothetical combinations, speculative extensibility, or impossible states.
- **Top-model subtraction pass before Phase 4 (no extra spawn):** remove every task/file/abstraction without an `AC-N`, collapse avoidable indirection, and confirm that task/file count did not grow beyond the approved behavior plus `required` constraints. After a review revision, rewrite the chosen plan coherently and delete superseded material — never append patches until the plan becomes a history log.
- **(conditional dual-plan only) Considered alternatives:** record the losing approach + selecting trade-off only when the material-fork trigger above ran (→ reference.md "Understand & research").
