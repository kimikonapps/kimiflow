## 🟡 Phase 4 — Plan-gate (binary; 2/3 rounds per strategy epoch)

Read → reference.md "Review rubric".

0. **Mechanical plan-blocker gate (before round 1):** run `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/plan-blocker-gate.sh .kimiflow/<slug>` (Codex: `KIMIFLOW_HOST=codex` and `KIMIFLOW_PLUGIN_ROOT`). `OPEN` is required before reviewers. For phase-read runs, record reads through Phase 4. It rechecks intent and Discovery completeness before plan executability, so do not spend reviewers on unresolved decisions/research, unmapped ACs, missing verification/path evidence, or undeclared affected files. Audit keeps its existing profile. Then check every AC maps to a task/test and no orphan task exists.
1. Spawn **2 independent `top`/`cross_family_top` reviewers in parallel** (scope=small → 1, lens B), fresh context, seeing only `PLAN.md` + `ACCEPTANCE.md` + `INTENT.md`/`PROBLEM.md` + named code (add `RESEARCH.md`/`DIAGNOSIS.md` only if the plan cites it; audit mode: `AUDIT-INTENT.md` + `AUDIT.md` + named code instead). `balanced`/`cheap` models never issue the plan-gate verdict. Frame each adversarially, but bind findings to evidence: a BLOCKER/HIGH requires a cited AC/intent boundary, `required` research constraint, current API/compatibility rule, project standard, or concrete security/data-loss failure with demonstrable impact. A plausible enhancement, extra robustness, hypothetical future case, or `optional` research item is never a blocking finding. Before adding anything, try to subtract unsupported tasks, files, abstractions, tests, and nesting. **Reviewers do not read reference.md:** inline into each spawn prompt (verbatim, <15 lines) the reviewer's lens definition + this evidence/scope rule, the FINDING grammar with its file-form constraints (one canonical line per finding, at column 0, no newline in the reason; nothing but FINDING lines, or the single sentinel line `NONE`), and the findings-file path. One lens is **cross-family by default** when a different-family CLI is available (`small` → the single lens-B reviewer; `large` → lens B) — transport, verbatim findings persist, sticky fallback → reference.md "Model routing (per-role)".
   - **A — goal/completeness & understanding** (goal-backward; canonical definition → reference.md "Review rubric").
   - **B — risk & subtraction:** concrete security/edge/error/architecture risk, cause-not-symptom in fix mode, and over-engineering removal; it does not invent future requirements (→ rubric).
   - **(audit) refute the cut:** for each `delete`/`yagni` slice, actively hunt a **live caller** (repo-wide, incl. dynamic dispatch / reflection / string-keyed lookup). A cut survives only if no reviewer finds one; any live caller → downgrade or move to do-NOT-touch. `shrink`/`stdlib` must preserve behavior (tests green before+after).
   - Each reviewer gives reasoning before verdict, then writes this round's findings to `.kimiflow/<slug>/findings/r<N>-<lens>.md` in the canonical one-line format (`FINDING <SEVERITY> <ref> :: <reason>`, sentinel `NONE` if clean — → reference.md "Review rubric"). No self-reported count; the orchestrator reads these files and never edits them.
2. Append a human-readable round summary to `REVIEW.md` (narrative only — reasoning, not the gate truth).
3. **Gate (binary, NO numeric score; strategy epochs):** each gate's rounds are global, monotonically increasing, and grammar-valid findings files are immutable. Let epoch start=`S`, round budget=`B` (`small`=2; `large`/audit=3), absolute cap=`C=S+B-1`; first epoch `S=1`. Persist `Review epoch`, `Review epoch start`, `Review epoch cap`, `Strategy fingerprint`, and `Recovery: active|clean` in STATE. Resolve via `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/resolve-review-gate.sh .kimiflow/<slug>/findings --round <N> --expect <lensCSV> --epoch-start <S> --cap <C>`. Only `OPEN/clean` opens; MEDIUM/LOW remain advisory. A finding resolves only when the next fresh reviewer no longer emits it.
4. **Automatic verdict routing:** `incomplete|malformed` → repair/substitute the reviewer seat and complete the same round (only grammar-invalid, never-counted files may be replaced per reference.md); `open-findings` → smallest evidenced repair + checks + next round inside the epoch; `oscillation|reappeared|cap-reached` → keep the gate CLOSED and enter recovery automatically. Never downgrade, auto-proceed, call `await-user`, or ask permission for another technical epoch.
5. **Recovery (compact, evidence-first):** append one epoch entry to `RECOVERY.md`: gate/trigger/source round/next `S`+`C`, blocker identities, failed strategy + refuting evidence, new falsifiable hypothesis, semantic strategy delta, before/after basis fingerprints, outcome. Re-read the cited code and AC/intent boundary; query run history/project memory once for the blocker+failed strategy (`--max 5`); use focused current primary-source research only for uncovered/stale facts; run the smallest refuting spike/check; then change hypothesis/algorithm/integration/architecture or AC-preserving task decomposition. Re-run the owning Discovery/diagnosis/plan checks, plan-blocker gate, AC mapping, and subtraction before the next round. Rewording/model-switch/file churn is not a strategy change; fingerprint change is necessary, not sufficient. After two failed recovery epochs, use one independent `top|cross_family_top` recovery solver, not extra standing reviewers. Technical blockers continue through `active-run.sh stop-gate`; only missing product authority/access, paid/privacy-sensitive infrastructure, scope/risk expansion, or an irreversible public/data/migration decision may use `await-user`. Existing one-time Preview/Commit gates are not repeated by recovery alone. → reference.md "Review rubric".
6. **Gate open →** `--prepare`: STOP, update STATE (0–4 done, `Status: backlog`), output `/kimiflow --resume <slug>`. Else → step 7.
7. **One pre-build Human Gate, mode-specific.** **Schema-3 fix:** show one plain-language **Fix Preview** from `PROBLEM.md` + proven `DIAGNOSIS.md` + the internally gated plan: **Verified cause · Will fix · Not included · Affected scope · Risks/regression**. Ask approve/change/defer. On approve, run `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/clarify-gate.sh .kimiflow/<slug> --record-fix-approval`, then require `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/clarify-gate.sh .kimiflow/<slug> --post-diagnosis` to return `OPEN` before Phase 5. The recorder writes `kimiflow:fix-approval` with a fingerprint of the approved problem/diagnosis/plan/acceptance/state basis; any later cause/fix/plan/scope/risk change closes the gate and requires a fresh Preview. This replaces the generic Build Preview/Risk Gate for fixes, including `full`; never ask twice. Defer/headless → backlog + resume. Schema <3 resumes its prior behavior. **Feature/audit:** derive the one-screen Build Preview from confirmed intent, Discovery decisions, and ACs: **Will build · Not included · Important decisions · Risks/irreversibility · Effort**. Record `Build risk: none|required` plus reason, then run `BG decide --state .kimiflow/<slug>/STATE.md --interactive <yes|no> [--alias full]`, where `BG=${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/resolve-build-gate.sh`. `CONTINUE` → Phase 5; `STOP` → approve/change/defer; `PARK` → backlog. Policy defaults `risk`; `always` always stops; `off` preview-only; legacy `on` maps `always`; feature/audit `full` always stops. `--prepare` still parks at step 6. → reference.md "Build Preview / Risk Gate".
