## 🟢 Phase 7 — Code-review ensemble against specs → fix → commit-gate

1. **Fixed review basis + compact packet.** Before any spawn, resolve `review_target_sha` with `git rev-parse HEAD`, then validate and pin one basis for the whole round: user-supplied base ref when present; otherwise the repository default branch for committed branch work; `HEAD` for working-tree-only review. Set `review_base_sha` to its merge-base with `review_target_sha` (or the target SHA for working-tree-only review), verify that the committed/worktree diff is non-empty, then record both refs/SHAs, exact diff commands, and reviewed commits in `CODE-REVIEW.md`; a clean diff ends without reviewer calls, and every reviewer receives the same basis. Discover and record `spec_sources` in precedence order (explicit user source → run-local `ACCEPTANCE.md` + `INTENT.md`/`PROBLEM.md` → issue/PR refs in reviewed commits → matching `docs/`, `specs/`, `.scratch/` material) and `standards_sources` (`AGENTS.md`/`CLAUDE.md`, `CONTRIBUTING.md`, `CODING_STANDARDS.md`, `.kimiflow/STANDARDS.md`, relevant architecture docs). If no spec exists, mark the Spec part unavailable instead of inventing intent; the correctness/regression part still runs against observable contracts. Build one compact packet: source references + only relevant excerpts, changed-file list, `git diff --stat`, relevant hunks, diagnosis/repro artifacts when present, decisive Phase-6 evidence, and exact verification commands.
2. **Candidate axes in parallel.** Read → reference.md "Review rubric" (orchestrator only). Spawn narrow fresh-context `top`/`cross_family_top` reviewers with adversarial framing; `balanced`/`cheap` models never issue semantic review verdicts. **Reviewers do not read reference.md** — inline each spawn's axis line + the CANDIDATE grammar + file-form constraints (column 0, no newline in the claim; nothing but CANDIDATE lines or the single sentinel `NONE`) into the prompt. `small` runs use at least **two** axes (`quick` alias: review light — ONE `spec-correctness` axis); add the third when the diff touches hooks, plugins, memory, launcher, APIs/contracts, multiple surfaces, or any high-risk path. On `small`, fold documented standards into R2 only when R3 is not scheduled; `large`/release-critical uses all three. One axis (default `spec-correctness`) is **cross-family by default** when available — same transport/fallback as Phase 4 (→ reference.md "Model routing (per-role)"). The standard axes are:
   - **R1 spec-correctness:** trace requirements independently; find missing/partial/wrong behavior, unrequested scope, logic/edge regressions, and weakened/insufficient tests.
   - **R2 failure-security:** input validation, secrets/privacy, path handling, rollback/failure atomicity, partial writes; on `small` without R3, also apply the documented-standards/smell dimension.
   - **R3 standards-integration:** documented project standards plus installed hooks/metadata/docs/launcher wiring, host parity (Claude/Codex), command/API/schema contracts; apply the compact smell baseline only as a heuristic.
   Each axis writes `.kimiflow/<slug>/code-review-candidates/r<N>-<axis>.md` using `CANDIDATE <SEVERITY> <ref> :: <claim> :: verify=<smallest check>` or `NONE`.
3. **Orchestrator verification, axis-preserving.** Verify each candidate with targeted code reads, commands, or reproduction; for BLOCKER/HIGH candidates include an **active refutation attempt** (execute its `verify=` check, read the full code path — "could this be wrong?"): refuted → rejected, not promoted. Write `CODE-REVIEW.md` under separate `Spec / Correctness`, `Failure / Security`, and `Standards / Integration` headings with source status, candidates, evidence, and accepted/rejected/unverified outcomes. Keep cross-axis duplicates visible and linked in the report; do not rerank across axes, but promote an exact underlying defect only once with all applicable axis labels so the mechanical count is not inflated. Only confirmed BLOCKER/HIGH/MEDIUM/LOW lines are promoted to `.kimiflow/<slug>/findings/r<N>-code-verified.md` as `FINDING <SEVERITY> <ref> :: [<axis-labels>] <reason>`, using `spec`, `risk`, and/or `standards` joined by `+`; if none are confirmed, write `NONE`. The resolver gate counts only this promoted file, never raw candidates.
4. **Advisories.** First stage the run's named changed paths (`git add <named paths>` — never `-A`; the scans read the staged diff). Then run the bundled test-weakening scan `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/test-weakening-scan.sh` over the staged diff → append its `FLAG` lines to `.kimiflow/<slug>/ADVISORIES.md`; if it can't be located or run, note it in STATE — never silently skip the advisory channel. Likewise run `hooks/secret-content-scan.sh` and `hooks/lsp-diagnostics.sh` when present. A smell with neither a documented-standard violation nor demonstrable impact is advisory-only; skip issues enforced by existing tooling. The Simplicity dimension still writes `FLAG`s to advisories (folded into an existing lens for `small`; dedicated prosecutor for `large` or size-tripwire).
5. **Fix + autonomous recovery.** Fix confirmed BLOCKER/HIGH, then re-run affected axes and repeat verification; carry forward every still-applicable verified finding from an unaffected axis. Shared/uncertain changes rerun all scheduled axes. Use Phase 4's exact strategy-epoch contract with gate=`code`: `S`, budget `B` (`quick`/`small`=2; `large`/release-critical=3), absolute `C=S+B-1`, globally increasing code-gate rounds, and `resolve-review-gate.sh ... --round <N> --expect code-verified --epoch-start <S> --cap <C>`. `incomplete|malformed` repairs transport in the same round; `open-findings` gets the next targeted repair; `oscillation|reappeared|cap-reached` writes a code entry to `RECOVERY.md`, returns to diagnosis/implementation/verification, preserves fix Red/Green evidence, uses bounded memory/current primary sources plus the smallest refuting check, changes strategy, and starts the next epoch without `await-user`. After two failed recovery epochs, use one independent recovery solver. Never waive a finding, reset/overwrite valid rounds, repeat build approval inside confirmed scope/risk, or bypass the final Commit Gate. → reference.md "Review rubric".
6. **Commit-gate — STOP.** Read → reference.md "Commit hygiene". **Advisory triage (fail-closed):** present every `.kimiflow/<slug>/ADVISORIES.md` `FLAG`; the commit is blocked until each is explicitly dismissed with a reason (legit refactor) or promoted (→ a real `FINDING HIGH` → back to phase-7 review). Then show: short summary, `git status`, `git diff --staged`. At `quiet`, summarize the diff in a few decisive sentences and point to the staged diff/status; do not paste reviewer reasoning or long diff excerpts. **Wait for explicit OK.** Then commit (only named paths, no `git add -A`, no co-author/AI trailer, tests green). **(large)** if `.kimiflow/test-gate` doesn't exist, write it with the test command verified green in phase 6 (idempotent, kept local/untracked — never staged or committed; suggest gitignoring `.kimiflow/`) and announce it. **(audit)** execute one slice at a time: verify its repo-wide grep returns 0 (A3), apply the cut/shrink, run the slice's verify gate (grep-sweep → typecheck/build → tests green; `shrink`/`stdlib` green before+after), edit companion tests in lockstep, then commit **one slice = one reviewable diff = one commit**. Never batch slices into one commit.
7. **Learning loop + project memory — required before `Status: done`.** Run `MR review-run --run .kimiflow/<slug> --write` after verification/review — it records the learning set in `.kimiflow/project/LEARNINGS.jsonl`, refreshes bounded memory/index/metrics/proposal/provider state, and returns a compact notification. `review-run` is **quality-gated**: too-short/generic learnings, missing evidence, or unsafe memory content block the write. If the run has no durable learning, use `MR review-run --run .kimiflow/<slug> --write --skip "<reason>"` so the skip is explicit. Then run `MR verify-run --run .kimiflow/<slug>`; a CLOSED result blocks completion, including stale/changed evidence fingerprints. User/workflow preferences go through `record --scope user` (local-only). Provider/Vault safety: neither `.kimiflow/` nor repo files store the key; prefetch/sync handoffs export only freshly verified current, non-private, non-security learnings; it never patches skills or writes external notes blindly — approved skill candidates become review-only drafts under `.kimiflow/project/SKILL-DRAFTS/`. Everything else (history search, usage/economics/usefulness metrics, `consolidate`/`propose` curation, the provider subcommands): → reference.md "Memory Router & Learning Loop".
8. **Project-map auto-refresh** (recommended, non-blocking). After Phase-6 verify, run `PMS refresh --changed` (`PMS` = `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/project-map-status.sh`; Codex: `KIMIFLOW_HOST=codex` + `KIMIFLOW_PLUGIN_ROOT`) so the sections of the files this run touched are re-hashed back to `current`, deleted members are pruned, new files under a known section prefix are adopted (+sha256), and each refreshed `.sh` section's symbol index is rebuilt. `NEW-FILE` lines are structure hints only — no auto-facts; durable learnings still flow through the gated learning loop above. Skip silently when there is no `.kimiflow/project/INDEX.json`. The Stop-hook map-staleness nudge (`hooks/map-staleness-nudge.sh`) is the safety net that surfaces any residual staleness. → reference.md "Project Map Bootstrap".
9. **Close state.** For phase-read runs, read/record the Phase 7 file before the final `hooks/active-run.sh finish --write`; finish checks fresh reads through Phase 7. Only after the commit gate and learning review are open, set all phases done and set `Status: done` in `STATE.md`.
