## 🔵 Phase 1 — Clarify (plain language): Intent (feature) or Problem (fix)

Goal: confirmed product intent for features/audits, and a usable problem brief for fixes, BEFORE research/plan. Use plain language, WHAT/WHY not HOW, and ask only for irreducible information. Features have no minimum question count: a complete request needs one compact confirmation. A clear bug report does **not** stop for a pre-diagnosis confirmation; ask only when missing symptom, expected behavior, or reproduction context blocks diagnosis. Trivial exact work may skip. → reference.md "Intent clarification" / "Fix mode".

- **Feature → intent clarification:** clarify goal, value, in/out of scope, "what done looks like" → write `INTENT.md` → **gate** "Does this match?" (OK to continue).
- **Fix → working problem brief:** capture symptom, expected vs. actual, and available reproduction context in `PROBLEM.md`; inspect and diagnose immediately when sufficient. No routine Human Gate here. The single pre-build Fix Preview moves to Phase 4 after the cause is proven.
- **Audit → scope clarification:** which paths, how aggressive, behavior-preserve constraints, do-NOT-touch hints, "what stays untouched" → write `AUDIT-INTENT.md` (plain language) → **gate** "Is this the right cleanup scope?" (OK to continue).
- **Mechanical clarify gate:** before Phase 2, run `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/clarify-gate.sh .kimiflow/<slug>` (Codex: `KIMIFLOW_HOST=codex` and `KIMIFLOW_PLUGIN_ROOT`). `OPEN` is required. For phase-read runs, first record fresh Phase 0/1 reads. Feature/audit `small|quick` artifacts carry `<!-- kimiflow:clarify-evidence behavior=confirmed scope=confirmed outcome=confirmed source=current-run -->`; fix mode requires the non-empty problem brief but no confirmation marker. The gate checks completeness/current confirmation, never a question count. Phase 4 runs its post-diagnosis form for new fix runs.
