## 🔵 Phase 1 — Clarify (plain language): Intent (feature) or Problem (fix)

Goal: source-backed product intent for features and a usable problem brief for fixes BEFORE research/plan. The user owns WHAT/WHY: goal, actor, visible behavior, boundaries, success, and material product/policy consequences. The agent owns HOW: architecture, dependencies, data model, internal APIs, code structure, tests, migrations, performance/concurrency mechanism, and implementation order. Never ask the user to design the solution. → reference.md "Intent clarification" / "Fix mode".

- **Coverage scan first:** inspect the request, code, project docs, tests, memory, and current sources. Mark each product dimension with `user_explicit|user_confirmed|project_evidence|reversible_default|not_applicable|unknown_material`; goal, behavior, and success require user/evidence provenance, never a default. Translate a necessary technical-looking uncertainty into its product consequence (for example "must it work offline?", never "which database?").
- **Selective elicitation:** rank only `unknown_material` product facts by Impact x Uncertainty and ask them in one compact batch. Budgets: `quick` ≤2, `small` ≤3, `large`/critical ≤5. Offer a recommended product default/choices; "I don't know" selects the smallest safe reversible default, with risky paid/privacy/irreversible behavior excluded unless explicitly accepted. Ask zero questions when coverage is already sufficient and never ask from a quota.
- **Intent Critic:** for `large`/critical features, run one bounded fresh-context critic over the request + compact coverage draft (≤900 input words, output only `COVERAGE_OK` or ≤5 missing user-owned facts; no HOW). Record `critic=passed`. `small|quick` folds the scan into the top orchestrator and records `critic=folded`; when fresh-agent routing is unavailable, record the fallback and let the top orchestrator run the identical critic packet before recording the pass.
- **One interaction, then autonomy:** after the optional batch answer, write the short Goal/Included/Excluded/Done contract and continue under current build authority. The answer establishes the contract; do not ask "Does this match?" or create a second routine confirmation. Technical gaps move to Phase 2 and autonomous recovery.

- **Feature → intent clarification:** perform the provenance scan, ask at most one product-fact batch, write `INTENT.md`, and show the simple contract. A route that will actually build needs current implementation authority; `plan`/`grill` stays no-code and never asks for future build permission.
- **Fix → working problem brief:** capture symptom, expected vs. actual, and available reproduction context in `PROBLEM.md`; inspect and diagnose immediately when sufficient. No routine Human Gate here. After the cause is proven, only a material product/authority/risk decision may pause before the bounded fix.
- **Audit → scope clarification:** capture paths, aggression, behavior-preserve constraints, do-NOT-touch hints, and what stays untouched → write `AUDIT-INTENT.md`. Discovery stays read-only and requests no cleanup authority; only a later user-selected fix slice may authorize implementation.
- **Mechanical clarify gate:** before Phase 2, run `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/clarify-gate.sh .kimiflow/<slug>` (Codex: `KIMIFLOW_HOST=codex` and `KIMIFLOW_PLUGIN_ROOT`). `OPEN` is required. For phase-read runs, first record fresh Phase 0/1 reads. New Contract-2 features carry the exact provenance marker from reference.md with `unknown_material=0`, `question_rounds=0|1`, `technical_questions=0`, current source, summary, critic receipt, and current build authority when applicable. The gate rejects generic `confirmed` self-certification. Runs without Contract 2, audits, fixes, and schema 3 retain their prior compatible contracts.
