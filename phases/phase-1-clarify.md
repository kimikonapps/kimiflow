## 🔵 Phase 1 — Clarify (plain language): Intent (feature) or Problem (fix)

Goal: confirmed product intent for features/audits, and a usable problem brief for fixes, BEFORE research/plan. Ask one compact batch only for missing facts that materially change product behavior, scope, UX, or acceptance; never interrogate about reversible technical HOW. Then show a short plain-language contract (goal, included, excluded, done) and continue when the current request already authorizes implementation. Ask for confirmation only when authority or a material choice is actually missing. A clear bug report proceeds directly to diagnosis. Trivial exact work may skip. → reference.md "Intent clarification" / "Fix mode".

- **Feature → intent clarification:** clarify goal, value, in/out of scope, and "what done looks like" → write `INTENT.md` → show the simple contract. A route that will actually build needs current implementation authority; `plan`/`grill` stays no-code and never asks for future build permission.
- **Fix → working problem brief:** capture symptom, expected vs. actual, and available reproduction context in `PROBLEM.md`; inspect and diagnose immediately when sufficient. No routine Human Gate here. After the cause is proven, only a material product/authority/risk decision may pause before the bounded fix.
- **Audit → scope clarification:** capture paths, aggression, behavior-preserve constraints, do-NOT-touch hints, and what stays untouched → write `AUDIT-INTENT.md`. Discovery stays read-only and requests no cleanup authority; only a later user-selected fix slice may authorize implementation.
- **Mechanical clarify gate:** before Phase 2, run `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/clarify-gate.sh .kimiflow/<slug>` (Codex: `KIMIFLOW_HOST=codex` and `KIMIFLOW_PLUGIN_ROOT`). `OPEN` is required. For phase-read runs, first record fresh Phase 0/1 reads. Every nontrivial schema-4 feature/audit carries `<!-- kimiflow:clarify-evidence behavior=confirmed scope=confirmed outcome=confirmed summary=present source=current-run -->`; actual feature-build routes additionally carry `authority=explicit|confirmed`, while no-code plan/audit discovery does not. Fix mode requires a non-empty problem brief. The gate checks completeness/current authority, never a question count. Schema 3 keeps its prior marker/Preview contract.
