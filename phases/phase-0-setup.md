## ⚪ Phase 0 — Setup, Routing & Scope-Gate

0. **Display verbosity preflight (first, before launcher/menu prose).** `RV` = `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/resolve-verbosity.sh`. Map any `--quiet`/`--verbose` to a level; `RV get [--flag <level>]` → the level that governs output immediately, including an empty-session launcher. Unset configuration resolves to `balanced` without a first-run dialog; customization is explicit through launcher/settings.
0.1. **Launcher routing (before slugging a normal run).** If the invocation is empty (`/kimiflow` / `$kimiflow` / `@kimiflow`), `--launcher`, `--menu`, or a vague explicit Kimiflow request without an actionable target, run `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/launcher-status.sh --pretty` (Codex: `KIMIFLOW_HOST=codex` and `KIMIFLOW_PLUGIN_ROOT`) and show the compact launcher menu from reference.md "Launcher mode" (one `.launcher.primary_action` + the compact status groups; `.launcher.maintenance.visible_reasons` front-door, `hidden_internal_reasons` drilldown-only). If resolved verbosity is `quiet`, show only the menu/status lines and one blocking question; no preamble, rationale, JSON summary, "I will" narration, or follow-up explanation. User picks a mode → route into it and continue Phase 0; "existing feature check" → `--verify-feature`; a backlog run → resume safety check before Phase 5; headless/no answer → STOP with the snapshot summary and do not auto-pick.
0.25. **Top-model preflight (before slugging a normal run).** Prefer the host's `top` model for orchestration and planning. When the host exposes model identity, verify it now: current Codex → `gpt-5.6-sol`; Claude → strongest available selected tier. A lower-tier session records a quality fallback and continues without asking the user to switch; use bounded top-tier review/verification seats when available, but never pretend they changed the active orchestrator. If model identity is unavailable, preserve the user's selected session and never intentionally downgrade orchestration. → reference.md "Model routing (per-role)".
0.5. **Git check + workspace inventory (first, before product clarification).** First run `git rev-parse --is-inside-work-tree`. No repo → skip the Git-only preflight, report it, and ask once whether to `git init` or run through verification only (phases 5–6, no commit). In a Git repo, run `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/workspace-preflight.sh status --pretty`, then show one plain summary: current root/branch/HEAD + dirty paths, worktree count, and each extra tree's dirty/ignored/locked/prunable/active/Kimiflow-owned/Codex-managed state. Default to the current worktree and create no implementation worktree. `prune --write` is allowed only for unregistered stale Git metadata; registered prunable metadata is unresolved. Never force remove, delete a branch, stash/reset/clean, or stage/commit foreign paths. Codex-managed trees are app-owned: offer their archive/retention controls, never alter them directly. Record any ambiguous dirty/foreign-active/manual-extra state for step 3.55; do not ask before the fresh schema-4 run and its active session exist. → reference.md "Workspace preflight".
1. **Slug + state dir.** Derive a kebab-case `<slug>`. State lives under `.kimiflow/<slug>/` at the git root of the current project — in plugin mode, creating `.kimiflow/` there also activates the `commit-secret-gate` hook for the repo (skill-only use loads no hook → reference.md "Commit hygiene").
2. **Mode routing — feature, fix, audit, feature-check, or natural alias.** First normalize a leading natural alias (`full|grill|plan|build|quick|review|audit|fix`) per the Modes table above (`full` = scope=`large`, not an approval trigger; `grill` stops after Phase 1; `plan` ≙ `--prepare`; `build` implements an approved prepared plan only). Then detect unresolved requests: build/add/change → feature; crashes/error/bug/wrong behavior → fix; remove dead code / over-engineering / "clean up" / "entschlacken" → **audit** (requires a target path — ask for one if missing); "already built" / "verify feature" / wiring checks → **feature-check** (requires a feature name/path or current diff — ask if missing). `--fix` / `--audit <path>` / `--verify-feature <feature-or-path>` still force the mode. In doubt, ask one simple question. Record mode, alias, and target in STATE. Fix mode branches phases 1–2 (`PROBLEM.md` ≙ `INTENT.md`, `DIAGNOSIS.md` ≙ `RESEARCH.md`) and uses the same schema-4 material-decision boundary. **Audit mode** branches phases 1–7 (→ reference.md "Audit mode"; always scope ≥ `small`). **Feature-check/review mode** is read-only (→ reference.md "Existing feature check"); it never enters Phase 5 unless the user selects a confirmed finding to fix as a new run.
3. **Resume check.** With `--resume <slug>`: read its `STATE.md`, resume at the first unfinished phase. Else if `.kimiflow/<slug>/STATE.md` exists → continue. Else create `STATE.md` with `Flow schema: 4`, feature/problem, slug, date, mode, scope, `Discovery required: yes` for a new non-trivial feature run (`no` for trivial/fix/audit/review), `Intent contract: 2` for a new non-trivial feature, `Architecture contract: 1`, `Architecture deliberation: pending`, a standalone `Affected files:` header followed by one repo-relative bullet per path once known, `Build risk: none|required` after Phase 2, `Status: active|backlog|done`, and `Phase 0..7`. Phase 2 resolves architecture deliberation to `off|active`; older runs without either contract stay resumable under their prior gate shape. Affected files are mandatory from Phase 3 and feed plan/staleness/frontend gates. Backlog means 0–4 done/5 open after a Phase-4 park; an earlier stop stays active. Runs without `Flow schema` predate versioned state; schemas 2–3 keep their prior resumable contracts.
3.5. **Active session lock.** For every normal write run, immediately run `hooks/active-run.sh start --run .kimiflow/<slug> --mode <mode> --scope <scope> --write` after `STATE.md` exists. If the returned status includes `phase_reads_required: true`, read `phases/PHASES.json`, read the Phase 0 file, then record it before step 3.55/Phase 1 with `hooks/active-run.sh phase-read --run .kimiflow/<slug> --phase 0 --file phases/phase-0-setup.md --write`; on entry to each later phase, read that phase's manifest file and immediately record it with the same command shape. Do not call `clarify-gate`, `plan-blocker-gate`, or `finish --write` until the required phase reads through that boundary are fresh. For follow-up user requests inside the same owner session, first add/update an item with `append-item` / `mark-built` / `mark-accepted` / `mark-rejected` / `drop-item`, then continue the current phase. Other sessions may read and plan; before shared-checkout writes they run `conflict-check --path <path>` for every intended path and proceed only on `allow_disjoint`; a Git worktree is exceptional, explicitly authorized, registered with `workspace-preflight.sh register`, and capped at one. If status reports `stale_risk=needs_revalidation`, revalidate and `refresh-baseline --write`. Schema 4 permits `await-user` only for `missing-input|authority|external-access|paid-privacy|scope-risk|irreversible|workspace`; routine preview/commit waits are invalid. Schema 3 retains its old typed waits for resumability. The Stop hook may block only the owner session until it finishes, parks, fails, or aborts the active run.
3.55. **Workspace disposition (still before product clarification).** If step 0.5 found ambiguity, call `active-run.sh await-user --run .kimiflow/<slug> --kind workspace --reason <batched-summary> --write` and ask **one batched upfront workspace decision** covering every item; STATE plus active-session receipts make a second workspace wait invalid on fresh, restarted, parked/resumed schema-4 runs. Apply only the selected safe actions, rerun status, and preserve uncertain provenance untouched. `remove --write` may retire only a registered noncurrent tree whose primary run is terminal (`done|failed|aborted`), clean (including ignored content), and unlocked; `parked` stays resumable. It pins and revalidates checkout plus Git administration, atomically archives the full checkout, then moves only the inode-matched administrative record into a reported metadata archive without `git worktree remove` or global prune, so cleanup cannot erase racing bytes or alter unrelated metadata. When status reports a safely eligible terminal leftover, the orchestrator invokes that guarded removal during this preflight instead of consuming the one-tree cap or prompting the user. Finally require `working-tree-gate.sh` OPEN for a fresh run, or a verified current-run-only dirty set for an already active run. If the selected workspace disposition advanced HEAD, immediately run `active-run.sh refresh-baseline --workspace-disposition --write`; it requires the one-shot workspace-decision receipt and a clean tree and pins that exact head for the frontend start. Explicitly authorized current-run paths may later be verified and atomically committed by name.
3.6. **Frontend selector (write runs only, no design payload).** New `feature|fix|audit` runs add exactly: `Frontend quality contract: 1`, `Frontend quality: off`, `Frontend quality routing: provisional`, `Frontend quality evidence: pending`, `Frontend quality basis: pending`, `Frontend quality start: pending`, `Frontend quality recovery: clean`, `Frontend quality recovery owns global: no`; keep global `Recovery: clean`. Only after step 3.55 has resolved workspace disposition and the working-tree gate, run `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/frontend-quality-gate.sh .kimiflow/<slug> --record-start --write` (Codex: `$KIMIFLOW_PLUGIN_ROOT/hooks/frontend-quality-gate.sh`) and require OPEN. It records the clean post-disposition frontend baseline/contract marker while preserving the Active-Run `started_head` as the original Phase-7 review base; it does not read a design reference. Read-only feature-check/review sets none of these fields.
4. **Phase task-list (glance widget).** Create one task per phase you will actually run (`TaskCreate`), scaled to scope (trivial → the few steps it runs; small/large → the phases of its loop). As you enter a phase set it `in_progress` (`TaskUpdate`) and `completed` when it closes. It **complements** STATE.md (durable/resume) and the colored markers (per-phase event line) and replaces narrated status — it does not change the engine. Subagents keep their OWN internal task-lists; do not mix them into the phase list.
5. **Git context assertion.** Reuse the step-0.5 result; re-run `git rev-parse --is-inside-work-tree` only if the working directory changed. Workspace preflight and local atomic commits require Git.
6. **Scope-gate — hard rule (the default protects simplicity-first):**
   - **Default = `small`** — `large` multiplies subagent/round (token) cost, so it's the exception, not a reflex. Bump to `large` only if ≥~5 files · new dependency or data migration · auth/security/money/privacy path · subtle/hard-to-reproduce bug · user asks for the full loop or uses the `full` alias. A run set to `large` includes the expected run ensemble in its one-line scope announcement (≈ up to N subagents across phases 2–7).
   - **`trivial`** = 1–2 files, no risk, and the user's requested output is exact enough that no user-visible behavior can be misread (fix: obvious cause, e.g. a typo).
   - In doubt, the smaller tier. Effect: trivial → no loop or intent round when exact. small → compact feature intent or direct fix diagnosis, adaptive discovery, 1 plan reviewer, sequential implementation, Phase-7 review ensemble. large / `full` → deeper understanding, strict plan-gate, risk-based decisions only, and the hard test-gate (written in Phase 7 from the Phase-6 command; → reference.md "Hard test-gate").
7. **Display verbosity / utility flags** (after launcher routing, before normal-run work; full rules → reference.md "Display verbosity").
   - **(a)** Reuse the `RV get [--flag <level>]` result from step 0; it already governs output.
   - **(b)** A normal actionable run never pauses for verbosity onboarding. Unset configuration uses `balanced`; surface customization later through the launcher/settings without interrupting the build.
   - **(c)** Explicit utility flags still act directly: `--set-verbosity <level>` → `RV set project`; `--settings` (dialog) → `RV set <scope>`. Write, report path, exit.
8. **Project Map Bootstrap / status check** (recommended, never a prerequisite; full rules → reference.md "Project Map Bootstrap"). `PMS` = `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/project-map-status.sh`. `--project-map quick` → create/update `.kimiflow/project/`, report paths, then STOP; `--project-map skip` → record `project_map: skipped` in `STATE.md` and continue. Otherwise, on non-trivial actionable runs: no `.kimiflow/project/INDEX.json` → record `project_map: skipped` and continue without asking; else run `PMS status`, record the result, and once likely affected paths are known run `PMS coverage --affected <path>...` and use its `phase2_depth=compressed|targeted|full` recommendation. Refresh a stale affected section automatically when the bounded local refresh is safe; otherwise record the gap and continue with direct code evidence, never a map-choice prompt. Explicit standalone map/doc/improve runs may still ask their material focus/storage question from reference.md (headless default `codebase+architecture` + `kimiflow` storage; repo-docs storage writes only publish-safe documentation from verified map facts).
