## ⚪ Phase 0 — Setup, Routing & Scope-Gate

0. **Display verbosity preflight (first, before launcher/menu prose).** `RV` = `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/resolve-verbosity.sh`. Map any `--quiet`/`--verbose` to a level; `RV get [--flag <level>]` → the level that governs output immediately, including an empty-session launcher. Do not run the first-run onboarding prompt before the launcher; if the invocation stops at the menu, no verbosity dialog fires.
0.1. **Launcher routing (before slugging a normal run).** If the invocation is empty (`/kimiflow` / `$kimiflow` / `@kimiflow`), `--launcher`, `--menu`, or a vague explicit Kimiflow request without an actionable target, run `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/launcher-status.sh --pretty` (Codex: `KIMIFLOW_HOST=codex` and `KIMIFLOW_PLUGIN_ROOT`) and show the compact launcher menu from reference.md "Launcher mode" (one `.launcher.primary_action` + the compact status groups; `.launcher.maintenance.visible_reasons` front-door, `hidden_internal_reasons` drilldown-only). If resolved verbosity is `quiet`, show only the menu/status lines and one blocking question; no preamble, rationale, JSON summary, "I will" narration, or follow-up explanation. User picks a mode → route into it and continue Phase 0; "existing feature check" → `--verify-feature`; a backlog run → resume safety check before Phase 5; headless/no answer → STOP with the snapshot summary and do not auto-pick.
0.25. **Top-model preflight (before slugging a normal run).** The active session must be the host's `top` model because it owns orchestration and planning. When the host exposes model identity, verify it now: current Codex → `gpt-5.6-sol`; Claude → strongest available selected tier. A lower-tier session → STOP and ask the user to switch; do not use a `top` subagent as a surrogate orchestrator. If model identity is unavailable, preserve the user's selected session and never intentionally downgrade orchestration. → reference.md "Model routing (per-role)".
0.5. **Working-tree start gate (normal write runs).** Before slugging any normal feature/fix/audit/project-map write run, run `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/working-tree-gate.sh` (Codex: `KIMIFLOW_HOST=codex` and `KIMIFLOW_PLUGIN_ROOT`). `OPEN` is required. If `CLOSED`, STOP before changing files and ask the user to commit, stash, or otherwise clean the worktree. Read-only launcher/status/verify-only inspection may still report the dirty state. `.kimiflow/` local state is ignored by the gate. → reference.md "Working-tree start gate".
1. **Slug + state dir.** Derive a kebab-case `<slug>`. State lives under `.kimiflow/<slug>/` at the git root of the current project — in plugin mode, creating `.kimiflow/` there also activates the `commit-secret-gate` hook for the repo (skill-only use loads no hook → reference.md "Commit hygiene").
2. **Mode routing — feature, fix, audit, feature-check, or natural alias.** First normalize a leading natural alias (`full|grill|plan|build|quick|review|audit|fix`) per the Modes table above (`full` = scope=`large` + one mode-specific Preview approval; `grill` stops after Phase 1; `plan` ≙ `--prepare`; `build` implements an approved prepared plan only). Then detect unresolved requests: build/add/change → feature; crashes/error/bug/wrong behavior → fix; remove dead code / over-engineering / "clean up" / "entschlacken" → **audit** (requires a target path — ask for one if missing); "already built" / "verify feature" / wiring checks → **feature-check** (requires a feature name/path or current diff — ask if missing). `--fix` / `--audit <path>` / `--verify-feature <feature-or-path>` still force the mode. In doubt, ask one simple question. Record mode, alias, and target in STATE. Fix mode branches phases 1, 2, and the Phase-4 Preview (`PROBLEM.md` ≙ `INTENT.md`, `DIAGNOSIS.md` ≙ `RESEARCH.md`). **Audit mode** branches phases 1–7 (→ reference.md "Audit mode"; always scope ≥ `small`). **Feature-check/review mode** is read-only (→ reference.md "Existing feature check"); it never enters Phase 5 unless the user selects a confirmed finding to fix as a new run.
3. **Resume check.** With `--resume <slug>`: read its `STATE.md`, resume at the first unfinished phase. Else if `.kimiflow/<slug>/STATE.md` exists → continue. Else create `STATE.md` with `Flow schema: 3`, feature/problem, slug, date, mode, scope, `Discovery required: yes` for a new non-trivial feature run (`no` for trivial/fix/audit/review), `Affected files: <paths>` once known, `Build risk: none|required` after Phase 2, `Status: active|backlog|done`, and `Phase 0..7`. Affected files are mandatory from Phase 3 and feed plan/staleness gates. Backlog means 0–4 done/5 open after a Phase-4 park; an earlier stop stays active. Runs without `Flow schema` predate versioned state and remain legacy-resumable; schema 2 keeps its prior Discovery contract.
3.5. **Active session lock.** For every non-trivial normal run, immediately run `hooks/active-run.sh start --run .kimiflow/<slug> --mode <mode> --scope <scope> --write` after `STATE.md` exists. If the returned status includes `phase_reads_required: true`, read `phases/PHASES.json`, read the Phase 0 file, then record it before Phase 1 with `hooks/active-run.sh phase-read --run .kimiflow/<slug> --phase 0 --file phases/phase-0-setup.md --write`; on entry to each later phase, read that phase's manifest file and immediately record it with the same command shape. Do not call `clarify-gate`, `plan-blocker-gate`, or `finish --write` until the required phase reads through that boundary are fresh. For follow-up user requests inside the same owner session, first add/update an item with `append-item` / `mark-built` / `mark-accepted` / `mark-rejected` / `drop-item`, then continue the current phase. Other sessions may read and plan; before writing in the shared checkout they run `conflict-check --path <path>` for each intended path and proceed only on `allow_disjoint`, otherwise wait/narrow scope/use a worktree. If `active-run.sh status` reports `stale_risk=needs_revalidation`, revalidate the affected plan/code and run `refresh-baseline --write` before finishing. Every schema-3 user wait uses `await-user --kind`: `missing-input|authority` for clarification, `preview` for the one pre-build gate, `commit` for the final gate, or the named external/risk/irreversibility kind; active recovery rejects `preview|commit`. The Stop hook may block only the owner session until it finishes, parks, fails, or aborts the active run.
4. **Phase task-list (glance widget).** Create one task per phase you will actually run (`TaskCreate`), scaled to scope (trivial → the few steps it runs; small/large → the phases of its loop). As you enter a phase set it `in_progress` (`TaskUpdate`) and `completed` when it closes. It **complements** STATE.md (durable/resume) and the colored markers (per-phase event line) and replaces narrated status — it does not change the engine. Subagents keep their OWN internal task-lists; do not mix them into the phase list.
5. **Git check.** `git rev-parse --is-inside-work-tree`. No repo → report + ask: `git init`, or run through verification only (phases 5–6), no commit (7). Parallel build requires git.
6. **Scope-gate — hard rule (the default protects simplicity-first):**
   - **Default = `small`** — `large` multiplies subagent/round (token) cost, so it's the exception, not a reflex. Bump to `large` only if ≥~5 files · new dependency or data migration · auth/security/money/privacy path · subtle/hard-to-reproduce bug · user asks for the full loop or uses the `full` alias. A run set to `large` includes the expected run ensemble in its one-line scope announcement (≈ up to N subagents across phases 2–7).
   - **`trivial`** = 1–2 files, no risk, and the user's requested output is exact enough that no user-visible behavior can be misread (fix: obvious cause, e.g. a typo).
   - In doubt, the smaller tier. Effect: trivial → no loop or intent round when exact. small → compact feature intent or direct fix diagnosis, adaptive discovery, 1 plan reviewer, sequential implementation, Phase-7 review ensemble. large / `full` → deeper understanding, strict plan-gate, one mode-specific Preview approval, and the hard test-gate (written in Phase 7 from the Phase-6 command; → reference.md "Hard test-gate").
7. **Display verbosity onboarding / utility flags** (after launcher routing, before normal-run work; full rules → reference.md "Display verbosity").
   - **(a)** Reuse the `RV get [--flag <level>]` result from step 0; it already governs output.
   - **(b)** `RV onboard-check [--flag <level>]` → `ASK`|`SKIP` (mechanical). For a normal run, `ASK` ∧ interactive → ask once for level + save-scope, then `RV set <project|global> <level>`. `SKIP`, headless, or dismissed → keep (a)'s level, no write, no block.
   - **(c)** Utility flags: `--set-verbosity <level>` → `RV set project`; `--settings` (dialog) → `RV set <scope>`. Write, report path, exit.
8. **Project Map Bootstrap / status check** (recommended, never a prerequisite; full rules → reference.md "Project Map Bootstrap"). `PMS` = `${CLAUDE_PLUGIN_ROOT:-$CLAUDE_SKILL_DIR}/hooks/project-map-status.sh`. `--project-map quick` → create/update `.kimiflow/project/`, report paths, then STOP; `--project-map skip` → record `project_map: skipped` in `STATE.md` and continue. Otherwise, on non-trivial runs: no `.kimiflow/project/INDEX.json` → offer once (`quick` / `skip`); else run `PMS status`, record the result, and once likely affected paths are known run `PMS coverage --affected <path>...` and use its `phase2_depth=compressed|targeted|full` recommendation; a stale affected section → offer a targeted delta refresh. Accepted standalone map/doc/improve runs ask the focus/storage menus from reference.md (headless default `codebase+architecture` + `kimiflow` storage; repo-docs storage writes only publish-safe documentation from verified map facts). Decline, headless, no answer, `unknown`, or `skip` → continue unchanged. If accepted, run bootstrap/refresh before Phase 2.
